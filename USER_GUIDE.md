# Руководство пользователя Strat-tools Server Monitor

Strat-tools Server Monitor — это веб-панель наблюдения за сервером или Raspberry Pi, который выполняет роль gateway/router, VPN endpoint, split-routing узла или вспомогательного инфраструктурного сервера.

Документ предназначен для пользователя dashboard. Он объясняет, что показывает интерфейс, как читать показатели и что делать при типовых проблемах.

## 1. Что это за система

Strat-tools Server Monitor помогает быстро понять, в каком состоянии находится сервер и связанные с ним сетевые функции.

Система показывает:

- загрузку CPU;
- использование RAM;
- температуру процессора;
- сетевой трафик по интерфейсам;
- состояние VPN;
- идет ли маршрут по умолчанию через VPN;
- доступна ли routing table `direct`;
- когда обновлялся список RU-адресов;
- время последнего обновления данных на странице.

Главная задача dashboard — дать быстрый ответ на практические вопросы:

- сервер жив и отвечает?
- VPN поднят?
- трафик идет через нужный интерфейс?
- не перегружен ли CPU или RAM?
- не перегревается ли устройство?
- актуален ли список RU-адресов?
- есть ли заметная сетевая активность?

Важно: это оперативный мониторинг. Он помогает смотреть текущее состояние и недавнюю историю, но не является долгосрочным архивом аналитики.

## 2. Как открыть интерфейс

Откройте браузер на компьютере, планшете или телефоне в той же сети, где доступен сервер.

Адрес обычно выглядит так:

```text
http://SERVER_IP:8080
```

Пример:

```text
http://192.168.1.25:8080
```

Где `SERVER_IP` — IP-адрес сервера или Raspberry Pi.

После открытия страницы вы должны увидеть dashboard с заголовком `Gateway Telemetry` и брендом `Strat-tools`.

Если страница не открывается:

- проверьте, что устройство находится в той же сети;
- проверьте, правильно ли указан IP-адрес;
- проверьте, правильно ли указан порт, например `8080`;
- попробуйте открыть страницу с другого устройства;
- если ничего не помогает, обратитесь к администратору сервера.

## 3. Общая структура страницы

Интерфейс разделен на несколько понятных зон.

### Верхняя шапка

В верхней части находится бренд-блок `Strat-tools` и название панели `Gateway Telemetry`.

Рядом расположены элементы управления:

- `Period` — период просмотра графиков;
- `Aggregation` — шаг агрегации данных;
- `Interface` — сетевой интерфейс для графика `Network traffic`.
- `Help` — встроенная справка по использованию dashboard.

### Встроенная справка Help

Кнопка `Help` открывает справку прямо поверх dashboard.

В справке есть разделы:

- О системе;
- Как читать dashboard;
- Схема маршрутизации;
- Описание метрик;
- Работа с графиками;
- Статусы;
- Типовые сценарии;
- Проблемы и их интерпретация;
- Ограничения;
- FAQ.

Справку можно закрыть:

- кнопкой `X`;
- клавишей `Esc`;
- кликом по затемненной области вокруг окна справки.

На карточках и графиках также есть маленькие значки `i`. Они показывают короткие контекстные подсказки по конкретной метрике или графику.

Раздел `Схема маршрутизации` в справке показывает ту же gateway-схему, что и документация проекта. Это удобно, если нужно быстро сверить, какие интерфейсы отвечают за локальные сегменты, прямой российский маршрут и VPN-маршрут, не покидая dashboard.

### Summary-карточки

Верхний блок карточек показывает самые важные текущие показатели:

- `CPU`;
- `RAM`;
- `Temperature`;
- `VPN status`;
- `Route via VPN`;
- `RU list`.

Эти карточки нужны для быстрой оценки состояния сервера без детального анализа графиков.

### Блок Routing & VPN health

Блок `Routing & VPN health` показывает компактные индикаторы инфраструктурного состояния:

- `Server`;
- `VPN`;
- `Default route`;
- `Direct table`;
- `RU list`.

Это быстрый контрольный список: если здесь все в норме, базовые сетевые функции, скорее всего, работают штатно.

### Графики

Основная часть страницы содержит графики:

- `CPU / Load`;
- `RAM`;
- `Temperature`;
- `Network traffic`.

Графики помогают увидеть не только текущее значение, но и изменение показателей во времени.

### Нижняя служебная часть

Внизу страницы отображается техническая информация:

- название продукта `Strat-tools Server Monitor`;
- `Node` — наблюдаемый узел;
- текущий выбранный интерфейс и скорость RX/TX;
- `Last refresh` — когда страница последний раз обновила данные;
- версия интерфейса, например `Build: in-memory v1`.

## 4. Описание показателей

### CPU

`CPU` показывает текущую загрузку процессора в процентах.

Нормально:

- низкие и средние значения;
- кратковременные всплески при обновлении маршрутов, VPN или сетевых задачах.

Тревожно:

- значение долго держится выше 70%;
- значение приближается к 90-100%;
- одновременно растет задержка сети или web-интерфейс начинает отвечать медленно.

Что это может означать:

- сервер выполняет тяжелую задачу;
- идет активная маршрутизация или обработка трафика;
- завис процесс;
- устройство слишком слабое для текущей нагрузки.

### RAM

`RAM` показывает использование оперативной памяти в процентах.

Нормально:

- значение стабильно и не растет бесконечно;
- на Linux часть памяти может использоваться под cache, это не всегда проблема.

Тревожно:

- RAM долго держится выше 75-90%;
- после перезапуска сервисов память быстро снова заполняется;
- вместе с высокой RAM растет CPU и система становится медленной.

Что это может означать:

- запущено слишком много процессов;
- один из сервисов потребляет слишком много памяти;
- устройству не хватает RAM для текущих задач.

### Temperature

`Temperature` показывает температуру CPU в градусах Celsius.

Нормально:

- температура умеренная и не растет постоянно;
- на Raspberry Pi температура может меняться в зависимости от нагрузки и корпуса.

Тревожно:

- температура долго держится около 65-80 C;
- значение приближается к 80 C и выше;
- одновременно падает производительность.

Что это может означать:

- плохое охлаждение;
- высокая нагрузка на CPU;
- пыль, закрытый корпус или плохая вентиляция;
- Raspberry Pi может снижать частоту процессора из-за перегрева.

Если вместо значения отображается `n/a`, значит система не смогла получить температуру. Остальные показатели при этом могут работать нормально.

### VPN status

`VPN status` показывает, активен ли VPN-сервис.

Возможные значения:

| Значение | Что означает |
|---|---|
| `active` | VPN-сервис запущен |
| `inactive` | VPN-сервис не активен |
| `unknown` | dashboard не смог определить состояние |

Нормально:

- `VPN status: active`;
- рядом в деталях `ppp0 up`;
- отображается IP-адрес `ppp0`, если он доступен.

Тревожно:

- `VPN status: inactive`;
- `ppp0 down`;
- IP-адрес `ppp0` отсутствует.

Что это может означать:

- VPN-сервис остановлен;
- VPN-соединение не установилось;
- нет связи с VPN-сервером;
- неверные учетные данные или сетевые настройки VPN.

### Route via VPN

`Route via VPN` показывает, идет ли default route через интерфейс `ppp0`.

Проще говоря, это проверка: использует ли сервер VPN как основной путь для исходящего трафика.

Возможные значения:

| Значение | Что означает |
|---|---|
| `via ppp0` | маршрут по умолчанию идет через VPN |
| `not via VPN` | маршрут по умолчанию не идет через VPN |
| `unknown` | состояние не удалось определить |

Нормально:

- если сервер должен отправлять основной трафик через VPN, ожидается `via ppp0`.

Тревожно:

- `not via VPN`, если по схеме работы трафик должен идти через VPN.

Что это может означать:

- VPN поднят, но маршруты не переключились;
- route table настроена неправильно;
- VPN есть, но трафик идет напрямую через провайдера;
- split-routing работает не так, как ожидается.

### RU list

`RU list` показывает свежесть списка RU-адресов.

Этот список может использоваться для split-routing: например, одни адреса идут напрямую, другие через VPN.

Возможные состояния:

| Значение | Что означает |
|---|---|
| `fresh` | список недавно обновлялся |
| `aging` | список обновлялся, но уже не очень свежий |
| `stale` | список давно не обновлялся |
| `n/a` | дата обновления неизвестна |

Нормально:

- `fresh`;
- дата обновления выглядит актуальной.

Тревожно:

- `stale`;
- `n/a`, если список должен регулярно обновляться.

Что это может означать:

- автоматическое обновление списка не запускалось;
- файл с датой обновления отсутствует;
- задача обновления завершилась ошибкой;
- путь к файлу обновления настроен иначе.

### Network traffic

`Network traffic` показывает входящий и исходящий трафик выбранного сетевого интерфейса.

Обычно используются:

- `eth0` — основной Ethernet-интерфейс;
- `eth0.3` — VLAN-интерфейс, если он настроен;
- `ppp0` — VPN-интерфейс.

На графике:

- `RX` — входящий трафик;
- `TX` — исходящий трафик.

Нормально:

- трафик появляется во время использования сети;
- на `ppp0` видна активность, если через VPN идет передача данных;
- на неиспользуемом интерфейсе трафик может быть почти нулевым.

Тревожно:

- трафик отсутствует на интерфейсе, который должен активно использоваться;
- внезапно появился очень большой трафик без понятной причины;
- `ppp0` активен, но на нем нет трафика при ожидаемой VPN-активности.

## 5. Работа с графиками

### Выбор периода

Поле `Period` управляет тем, за какой промежуток времени показываются графики.

Доступные варианты:

| Period | Когда использовать |
|---|---|
| `1 hour` | посмотреть последние события и текущую ситуацию |
| `24 hours` | оценить поведение за день |
| `7 days` | увидеть общую динамику за неделю |

Если график кажется пустым после перезапуска сервера, это может быть нормально: история хранится только в памяти и набирается заново.

### Выбор агрегации

Поле `Aggregation` управляет тем, насколько подробно показываются точки графика.

Доступные варианты:

| Aggregation | Что означает |
|---|---|
| `raw` | сырые точки, максимальная детализация |
| `1 minute` | усреднение по 1 минуте |
| `5 minutes` | усреднение по 5 минутам |
| `15 minutes` | усреднение по 15 минутам |

Практическое правило:

- для периода `1 hour` удобно использовать `raw` или `1 minute`;
- для периода `24 hours` удобно использовать `1 minute` или `5 minutes`;
- для периода `7 days` удобно использовать `5 minutes` или `15 minutes`.

### Выбор интерфейса

Поле `Interface` влияет на график `Network traffic`.

Выберите:

- `eth0`, чтобы посмотреть основной сетевой интерфейс;
- `eth0.3`, чтобы посмотреть VLAN-интерфейс;
- `ppp0`, чтобы посмотреть VPN-трафик.

Если выбранный интерфейс не существует или сейчас не активен, график может быть пустым.

### Как читать CPU / Load

График `CPU / Load` показывает изменение CPU во времени.

Обратите внимание:

- разовые пики обычно не опасны;
- длительная высокая полка может означать проблему;
- если рост CPU совпадает с ростом сетевого трафика, нагрузка может быть связана с маршрутизацией.

### Как читать RAM

График `RAM` показывает, как менялось использование памяти.

Обратите внимание:

- плавный рост без снижения может указывать на утечку памяти или накопление нагрузки;
- стабильное значение обычно нормально;
- резкие скачки возможны при запуске сервисов.

### Как читать Temperature

График `Temperature` показывает динамику температуры.

Обратите внимание:

- температура часто растет вместе с CPU;
- если температура высокая даже при низком CPU, проверьте охлаждение;
- если график пустой, датчик температуры может быть недоступен.

### Как читать Network traffic

График `Network traffic` показывает `RX` и `TX` выбранного интерфейса.

Обратите внимание:

- `RX` и `TX` могут отличаться, это нормально;
- при активном VPN на `ppp0` обычно должен быть заметен трафик;
- отсутствие трафика не всегда ошибка: возможно, сеть просто не используется.

## 6. Статусы и индикаторы

Dashboard использует спокойные цветовые состояния.

| Цвет / статус | Значение | Как реагировать |
|---|---|---|
| Green / OK | Все выглядит нормально | Обычно действий не требуется |
| Amber / Warning | Есть потенциальная проблема или неизвестность | Стоит обратить внимание |
| Red / Bad / Critical | Есть явная проблема | Нужно проверить причину |
| Blue / Neutral / Informational | Информационный показатель | Используйте для наблюдения |

Важно: цвет помогает быстро сориентироваться, но окончательный вывод лучше делать по сочетанию нескольких показателей.

Например:

- `VPN active`, но `Route via VPN: not via VPN` — VPN работает, но маршрутизация может быть неправильной.
- `CPU` высокий и `Temperature` высокая — возможно, устройство перегружено и нагревается.
- `RU list stale`, но VPN работает — проблема не в VPN, а в обновлении списка адресов.

## 7. Типовые пользовательские сценарии

### Проверить, работает ли VPN

Посмотрите на карточку `VPN status`.

Нормальный признак:

- `active`;
- в деталях `ppp0 up`;
- есть IP-адрес `ppp0`.

Дополнительно посмотрите блок `Routing & VPN health`:

- `VPN: active` должен быть зеленым.

Если `VPN inactive` или `ppp0 down`, VPN сейчас не работает штатно.

### Понять, идет ли трафик через VPN

Посмотрите карточку `Route via VPN`.

Если там `via ppp0`, default route идет через VPN.

Затем выберите `Interface: ppp0` и посмотрите график `Network traffic`.

Если на `ppp0` есть RX/TX, значит через VPN-интерфейс идет передача данных.

Если `Route via VPN: not via VPN`, трафик может идти напрямую, даже если VPN-сервис активен.

### Увидеть рост нагрузки на CPU

Откройте график `CPU / Load`.

Выберите:

- `Period: 1 hour`, если проблема происходит сейчас;
- `Period: 24 hours`, если нужно понять, была ли нагрузка в течение дня.

Если CPU долго держится высоким, проверьте, совпадает ли это с ростом `Network traffic` или `Temperature`.

### Проверить, обновлялся ли список RU-адресов

Посмотрите карточку `RU list`.

Нормально:

- `fresh`;
- дата обновления недавняя.

Если `aging` или `stale`, список давно не обновлялся.

Если `n/a`, dashboard не видит дату обновления.

### Посмотреть, какой интерфейс нагружен сильнее

Используйте поле `Interface`.

По очереди выберите:

- `eth0`;
- `eth0.3`;
- `ppp0`.

Сравните график `Network traffic`.

Интерфейс с большими значениями RX/TX сейчас несет больше трафика.

## 8. Типовые проблемы и их интерпретация

### Страница не открывается

Как выглядит:

- браузер пишет, что сайт недоступен;
- страница долго грузится;
- ошибка подключения.

Что обычно означает:

- неверный IP или порт;
- сервер недоступен по сети;
- сервис мониторинга остановлен;
- доступ блокирует firewall.

Что можно сделать самостоятельно:

- проверить адрес в браузере;
- попробовать открыть с другого устройства в той же сети;
- проверить, подключен ли сервер к сети;
- подождать 1-2 минуты после перезагрузки сервера.

Когда обращаться к администратору:

- если страница не открывается с нескольких устройств;
- если сервер недавно перезагружался и dashboard не появился;
- если вы не знаете правильный IP или порт.

### VPN inactive / ppp0 down

Как выглядит:

- `VPN status: inactive`;
- `ppp0 down`;
- индикатор `VPN` красный или предупреждающий.

Что обычно означает:

- VPN-сервис не запущен;
- VPN-соединение разорвано;
- интерфейс `ppp0` не создан;
- есть проблема с доступом к VPN-серверу.

Что можно сделать самостоятельно:

- проверить, есть ли интернет на стороне сервера;
- посмотреть, повторяется ли проблема через несколько минут;
- если есть доступ только к dashboard, зафиксировать время проблемы.

Когда обращаться к администратору:

- если VPN нужен для работы сети;
- если статус не восстанавливается самостоятельно;
- если одновременно `Route via VPN` показывает `not via VPN`.

### Route via VPN: not via VPN

Как выглядит:

- карточка `Route via VPN` показывает `not via VPN`;
- в health-блоке `Default route` не зеленый.

Что обычно означает:

- default route не направлен через `ppp0`;
- VPN может быть активен, но маршрутизация не переключилась;
- split-routing работает не так, как ожидается.

Что можно сделать самостоятельно:

- проверить, активен ли `VPN status`;
- выбрать `Interface: ppp0` и посмотреть, есть ли трафик;
- сравнить активность `ppp0` и `eth0`.

Когда обращаться к администратору:

- если по правилам вашей сети основной трафик обязан идти через VPN;
- если сайты или сервисы стали открываться не так, как обычно.

### RU list stale

Как выглядит:

- карточка `RU list` показывает `stale`;
- индикатор `RU list` в health-блоке красный или предупреждающий.

Что обычно означает:

- список RU-адресов давно не обновлялся;
- задача обновления не запускалась;
- файл с датой обновления отсутствует или недоступен.

Что можно сделать самостоятельно:

- проверить дату, показанную в карточке;
- оценить, есть ли проблемы с доступом к сайтам, которые должны идти напрямую или через VPN.

Когда обращаться к администратору:

- если список остается `stale` больше суток;
- если после этого появились проблемы с маршрутизацией сайтов.

### Температура слишком высокая

Как выглядит:

- `Temperature` окрашена как предупреждение или проблема;
- график температуры держится на высоком уровне.

Что обычно означает:

- устройство перегревается;
- высокая нагрузка на CPU;
- плохое охлаждение или вентиляция.

Что можно сделать самостоятельно:

- убедиться, что устройство не закрыто и не лежит рядом с источником тепла;
- проверить, работает ли вентилятор, если он есть;
- посмотреть, совпадает ли высокая температура с высокой загрузкой CPU.

Когда обращаться к администратору:

- если температура долго держится высокой;
- если сервер начинает работать медленно;
- если устройство часто перезагружается.

### Один из графиков пустой

Как выглядит:

- на графике нет линий;
- значения отсутствуют или показываются как `n/a`.

Что обычно означает:

- сервис недавно запущен и история еще не накопилась;
- метрика недоступна на этом устройстве;
- выбранный сетевой интерфейс отсутствует или не активен;
- после перезапуска история была очищена.

Что можно сделать самостоятельно:

- подождать 1-2 минуты;
- переключить `Period` на `1 hour`;
- для сети выбрать другой `Interface`;
- обновить страницу.

Когда обращаться к администратору:

- если график остается пустым долго;
- если пустые сразу несколько важных графиков;
- если одновременно не обновляется `Last refresh`.

### Нет сетевой активности по интерфейсу

Как выглядит:

- на `Network traffic` линии RX/TX почти нулевые;
- выбран, например, `ppp0`, но график пустой.

Что обычно означает:

- через интерфейс сейчас нет трафика;
- интерфейс не используется;
- VPN не поднят;
- выбран не тот интерфейс.

Что можно сделать самостоятельно:

- выбрать другой интерфейс;
- проверить `VPN status`;
- проверить `Route via VPN`;
- открыть период `24 hours`, чтобы увидеть, была ли активность раньше.

## 9. Ограничения системы

Strat-tools Server Monitor предназначен для оперативного наблюдения, а не для долгосрочного хранения истории.

Ограничения:

- данные хранятся только в оперативной памяти процесса;
- история ограничена временным окном, выбранным в настройках системы;
- после рестарта сервиса или сервера графики могут стать пустыми;
- система не является архивом долгосрочной аналитики;
- dashboard не заменяет полноценную систему расследования инцидентов;
- часть метрик зависит от доступности системных интерфейсов и сервисов;
- если VPN, интерфейс или датчик температуры недоступны, соответствующие поля могут показывать `n/a`, `unknown` или пустой график.

Это сделано намеренно: система легкая, не требует базы данных и подходит для небольшого gateway/router-сервера.

## 10. Часто задаваемые вопросы

### Почему после перезагрузки графики пустые?

Потому что история хранится только в оперативной памяти. После перезапуска сервиса или сервера старые точки исчезают, и графики начинают заполняться заново.

### Почему нет данных по интерфейсу?

Возможные причины:

- интерфейс не существует на сервере;
- интерфейс сейчас отключен;
- через него нет трафика;
- вы выбрали не тот интерфейс в поле `Interface`;
- сервис недавно запустился и еще не накопил точки.

Попробуйте выбрать другой интерфейс и подождать 1-2 минуты.

### Почему VPN status зеленый, но трафик маленький?

`VPN status` показывает, что VPN-сервис активен. Это не означает, что через VPN прямо сейчас обязательно идет большой трафик.

Трафик может быть маленьким, если:

- сеть сейчас почти не используется;
- часть маршрутов идет напрямую;
- активные подключения не требуют VPN;
- выбран короткий период просмотра.

Для проверки выберите `Interface: ppp0` и посмотрите `Network traffic`.

### Что означает Default route via VPN?

Это проверка, идет ли основной маршрут исходящего трафика через VPN-интерфейс `ppp0`.

Если отображается `via ppp0`, сервер использует VPN как основной путь.

Если отображается `not via VPN`, основной путь идет не через VPN. В некоторых схемах split-routing это может быть нормально, но если весь трафик должен идти через VPN, это проблема.

### Зачем нужен RU list update?

RU list update показывает, когда последний раз обновлялся список RU-адресов.

Такой список может использоваться для split-routing: например, чтобы часть адресов шла напрямую, а часть через VPN.

Если список устарел, маршрутизация некоторых адресов может работать не так, как ожидается.

### Почему Temperature показывает n/a?

Dashboard не смог получить температуру CPU.

Это может быть нормально, если:

- устройство не предоставляет датчик температуры;
- датчик недоступен системе;
- команда чтения температуры не установлена;
- сервис не имеет доступа к источнику температуры.

Остальные показатели при этом могут работать корректно.

### Что означает Direct table?

`Direct table` показывает наличие routing table `direct`.

Она может использоваться в схемах split-routing, где часть трафика направляется напрямую, а часть через VPN.

Если `Direct table` показывает `missing`, это не всегда авария, но для схемы, где она обязательна, это признак проблемы маршрутизации.

### Как понять, что dashboard обновляется?

Посмотрите нижнюю строку `Last refresh`.

Если время меняется примерно каждые несколько секунд, страница получает новые данные.

Если время не меняется или появилось `Last refresh failed`, страница не смогла получить данные от сервера мониторинга.

### Можно ли использовать систему как архив статистики?

Нет. Система предназначена для оперативного мониторинга. Она не хранит долгосрочную историю на диске.

Для долгосрочного архива нужна отдельная система мониторинга, но она не входит в эту легкую версию dashboard.

## 11. Краткая памятка

Если нужно быстро проверить состояние gateway:

1. Откройте `http://SERVER_IP:8080`.
2. Посмотрите summary-карточки `CPU`, `RAM`, `Temperature`, `VPN status`, `Route via VPN`, `RU list`.
3. В блоке `Routing & VPN health` проверьте `VPN`, `Default route`, `Direct table`.
4. Для проверки VPN-трафика выберите `Interface: ppp0`.
5. Для текущей ситуации используйте `Period: 1 hour`.
6. Для дневной картины используйте `Period: 24 hours`.
7. Если графики пустые после рестарта, подождите несколько минут.
8. Если статусы красные или долго остаются предупреждающими, обратитесь к администратору.

## 12. Актуальные возможности dashboard

Этот раздел описывает элементы интерфейса, которые появились в текущей версии dashboard.

### Схема маршрутизации

В проект добавлена схема маршрутизации gateway. Она доступна в двух местах:

- во встроенной справке dashboard по кнопке `Help`, раздел `Схема маршрутизации`;
- в Markdown-документации проекта ниже.

![Описание маршрутизации gateway](Docs/assets/routing-overview.svg)

Подробное описание схемы находится в [Docs/routing-overview.md](Docs/routing-overview.md).

Схема показывает:

- откуда приходит трафик на `eth0` и `eth0.3`;
- как маршрутизатор `sgate` классифицирует российский и зарубежный трафик;
- почему российский трафик идет напрямую через `eth0`;
- почему зарубежный трафик идет через VPN-интерфейс `ppp0`;
- где применяется NAT masquerade;
- как обновление RU-списка связано с маршрутизацией.

### Подсказки `i`

На summary-карточках и графиках есть маленький значок `i`. При наведении или фокусе он показывает короткую темную подсказку.

Подсказки есть у:

- CPU;
- RAM;
- Temperature;
- VPN status;
- Route via VPN;
- RU list;
- CPU / Load graph;
- RAM graph;
- Temperature graph;
- Network traffic graph.

Подсказка отображается как один темный tooltip. Нативный светлый tooltip браузера не используется.

### Переключатель Traffic units

Поле `Traffic units` меняет только отображение сетевых значений:

- `Bytes` показывает трафик как `B/s`, `KB/s`, `MB/s`.
- `Bits` показывает трафик как `b/s`, `kb/s`, `Mb/s`.

Backend продолжает хранить и отдавать сетевую скорость в bytes/sec. Перевод в bits/sec выполняется только в браузере.

Переключатель влияет на:

- сетевые графики;
- tooltip сетевых графиков;
- peak statistics;
- нижнюю строку `Node`;
- подписи сетевых графиков.

Выбранный режим сохраняется в браузере и восстанавливается при следующем открытии dashboard.

### Дополнительные сетевые графики

Кнопка `Add network chart` находится в блоке выбора интерфейсов. Она создает дополнительный график сетевого трафика.

Для каждого сетевого графика можно:

- выбрать собственный набор интерфейсов;
- изменить название графика;
- скрывать отдельные RX/TX-линии через легенду графика;
- перетащить график в другую строку dashboard.

Базовый сетевой график удалить нельзя. Дополнительные графики можно удалить кнопкой `Remove` внутри карточки графика.

Настройки дополнительных графиков сохраняются в `localStorage` браузера. После перезагрузки страницы dashboard должен восстановить:

- список дополнительных сетевых графиков;
- выбранные интерфейсы каждого графика;
- пользовательские названия;
- порядок и строки размещения графиков;
- скрытые через легенду RX/TX-линии.

### Компоновка графиков

Графики можно перетаскивать мышью и раскладывать по строкам.

Строка может содержать:

- один график на всю ширину;
- два графика поровну;
- три графика в одной строке.

Чтобы создать новую строку, перетащите график в зону между строками. Layout сохраняется в браузере.

### Peak statistics

Блок `Peak traffic by interface` показывает максимальную зафиксированную скорость трафика по каждому наблюдаемому интерфейсу.

Для каждого интерфейса отображается:

- `Max RX`: максимальная входящая скорость;
- `Max TX`: максимальная исходящая скорость.

Максимумы считаются с момента старта сервиса или с момента последнего нажатия `Reset peak stats`.

Кнопка `Reset peak stats` сбрасывает все максимумы. После сброса значения начинают считаться заново.

Важно: peak statistics хранится только в памяти процесса. После перезапуска сервиса значения очищаются.
